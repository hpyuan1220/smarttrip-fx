# SmartTrip FX — 第一版 MVP PRD

> **一句話定位:** 不是「又一個 AI 行程產生器」,而是**「唯一會幫你算『該帶多少外幣現金、何時換最划算』的個人旅遊小工具」**,主打台灣 → 日韓的說走就走旅客。
>
> **文件狀態:** v0.1 草稿 · 多數功能已上線(<https://smarttrip-fx.vercel.app>) · 本 PRD 用來收斂 MVP 範圍與驗證指標。

---

## 1. 問題陳述 (Problem Statement)

台灣出境旅客(尤其去日本)最頭痛的不是「去哪玩」—— 這在 ChatGPT、Google、Klook 上已是免費商品 —— 而是**「我到底要換多少現金?哪些地方只收現金?現在換划不划算?」**。

現有行程規劃工具與 OTA 完全不處理這件事,旅客只能憑感覺換匯,結果常常**換太多**(回國再換回去虧匯差)或**換太少**(現場狼狽)。在一趟 NT$40,000 的旅程中,匯損與多換的成本可達數百到上千元。

## 2. 目標 (Goals)

- **G1(用戶價值):** 讓使用者在 **3 分鐘內**得到一份含「**精準建議換匯現金額**」的行程,且事後實際現金花費與建議值誤差 < 15%。
- **G2(核心假設驗證):** 證明「金錢優先」定位有需求 —— 達到指定留存/回訪門檻(見第 7 節)。
- **G3(差異化護城河):** 使用者在問卷/訪談中能自發說出「這個和別的行程 App 不一樣的地方是換匯/現金」(無提示 ≥ 60%)。
- **G4(內容信任):** 建議換匯額與 FX 燈號被認為「可信、夠用」(滿意度 ≥ 4/5)。

## 3. 非目標 (Non-Goals)

- **不做真實訂房/訂票或庫存比價** —— 用外部連結導出即可;自建或接 API 成本高、非驗證重點。
- **不做帳號/雲端同步** —— MVP 用本機 localStorage,保留「免登入、即開即用」的簡單性;跨裝置留待 V1。
- **不做個人化推薦/歷史學習** —— 資料量不足以有效個人化,屬 V2。
- **不做社群/協作行程** —— 非核心假設。
- **不做多人分帳** —— 開支紀錄先以單人視角。

## 4. 目標用戶 (Persona)

- **主要:小資衝動型出國客(25–40 歲,台灣)。** 一年 1–4 趟、以日韓泰為主,預算敏感、習慣帶現金、會看廉航/特價,決策快、怕麻煩。
- **次要:首次自由行新手。** 不知道該換多少錢、哪裡要現金,需要被「手把手」算好。

## 5. User Stories

### 規劃
- 作為衝動型旅客,我想**選心情 + 預算範圍就生出三種方案**,這樣我不用花時間細排就能決定。
- 作為旅客,我想**一眼看到「建議換匯現金 ¥XXXXX」**,這樣我去銀行/機場換錢時心裡有數。
- 作為旅客,我想知道**每個行程是收現金還是能刷卡**,這樣我不會在只收現金的店尷尬。
- 作為旅客,我想看到**現在換匯划不划算的燈號**,這樣我能決定要不要現在換。

### 紀錄與回顧
- 作為旅客,我想**把選好的方案存起來**,出發前隨時看。
- 作為旅客,我想在旅途中**隨手記開支並對比預算**,這樣我知道有沒有超支。

## 6. 需求 (Requirements)

### P0 — 必須(MVP 不能少,用來驗證核心假設)

| # | 需求 | 驗收標準 | 現況 |
|---|---|---|---|
| P0-1 | 規劃輸入:目的地、日期、消費/本國幣別、**預算範圍**、心情/主題/同行/人數 | Given 必填齊全 / When 按生成 / Then 回傳三級距結果;缺項有明確錯誤 | ✅ 已上線 |
| P0-2 | **Low/Mid/High 三方案**,各有總花費與建議換匯額 | 三方案總額遞增且各自 ≤ 該級距預算 | ✅ 已上線 |
| P0-3 | 行程時間軸,**每項標註現金/刷卡 + 預估花費** | 每張卡片皆有支付標籤與金額(依幣別格式化) | ✅ 已上線 |
| P0-4 | **建議換匯現金額** = cash_only 總和 × 1.1,依幣別進位 | 顯示金額、台幣約當、緩衝說明;數字與規則一致 | ✅ 已上線 |
| P0-5 | **FX 換匯燈號**(今日 vs MA30:STRONG_BUY/BUY/HOLD)+ 文字建議 | 顯示燈號、今日匯率、MA30、偏離 % | ⚠️ 已上線但**為模擬資料** |
| P0-6 | **儲存行程**到本機 | 存檔後於「我的行程」可見、可重開、重整不消失 | ✅ 已上線 |
| P0-7 | **開支紀錄**(現金/刷卡)+ **計畫 vs 實際**對比 | 新增/刪除開支即時更新總額、超支/結餘 | ✅ 已上線 |
| **P0-8** | **接入即時匯率**取代模擬值(讓「省錢/時機」可信) | 燈號與匯率來自真實來源;失敗時明確標示「模擬」 | ❌ **待補(關鍵)** |
| **P0-9** | **基本使用分析**(生成數、存檔率、回訪) | 事件可量測(見第 8 節) | ❌ **待補(關鍵)** |

> **觀點:** P0-8 與 P0-9 是目前缺口。沒有 P0-8,「省錢」是空話;沒有 P0-9,無法驗證假設。建議 MVP 收尾就補這兩項。

### P1 — 應該有(快速跟進,非阻擋上線)

- 機票 + 住宿**估算** + 訂票搜尋連結 + **撿便宜/最後一分鐘**情報(✅ 已上線)。
- **旅程回顧**(星等 + 心得)、**心情故事**(✅ 已上線)—— 增加回訪黏著,但非核心假設驗證所必需。
- 多目的地 + 30+ 幣別(✅ 已上線)。
- 行程/換匯結果**分享或匯出**(連結/截圖/PDF)—— 利於口碑擴散。

### P2 — 未來(先不做,但架構預留)

- 帳號 + 雲端同步(跨裝置)。
- 「確認所選機票/飯店」把估算換成真實紀錄。
- 轉寄訂房信自動記帳 → 累積個人歷史。
- 個人化推薦(金錢習慣優先,如「你都換太多」)。
- 聯盟分潤 / 即時匯率到價提醒(訂閱)。

## 7. 成功指標 (Success Metrics)

> 以下為**假設值**,需 P0-9 上線後校準(列為 Open Question)。

### 領先指標(天 ~ 週)
- **啟用率:** 進站者完成一次生成 ≥ **40%**(stretch 55%)。
- **存檔率:** 生成後按「儲存此行程」 ≥ **25%**。
- **完成時間:** 從進站到拿到結果中位數 < **3 分鐘**。
- **外部連結點擊率:** 生成者中 ≥ **30%** 點任一搜尋/特價連結(代表真實出行意圖)。

### 落後指標(週 ~ 月)
- **回訪:** 30 天內回來再生成或開啟已存行程 ≥ **20%**。
- **價值證明:** 有記開支的行程中,**實際現金 vs 建議換匯誤差 < 15%** 的比例 ≥ **60%**(證明換匯算得準)。
- **差異化認知:** 訪談/問卷無提示說出「換匯/現金」差異 ≥ **60%**;整體滿意度 ≥ **4/5**。

**判定:** 任一驗證窗(建議 4 週)達啟用率 + 回訪雙門檻 → 假設成立,進 V1(即時匯率深化 + 帳號 + 分潤)。否則重新檢視定位。

## 8. 量測方式 (待建)

- 導入輕量分析(如 Plausible / GA4 / PostHog),事件:`generate`、`tier_select`、`save_trip`、`add_expense`、`deal_link_click`、`return_visit`。
- 「換匯準確度」需在開支紀錄計算 `實際現金 / 建議換匯` 並(匿名)回報。

## 9. Open Questions

- **(資料)** 即時匯率資料源選哪個?成本/額度/涵蓋幣別?(P0-8 阻擋項)
- **(資料/隱私)** 要不要做匿名分析?在「免登入、重隱私」定位下如何取得同意?(P0-9)
- **(產品)** MVP 是否需要「換匯準確度」回饋迴圈才算完整,還是 V1 再做?
- **(商業)** 驗證後優先走**聯盟分潤**還是**即時匯率訂閱**?(影響 V1 排序)
- **(指標)** 上述目標數字無歷史基準,需首兩週實測後重設。
- **(法務)** FX 燈號/換匯建議是否需要「非投資/理財建議」免責聲明強化?

## 10. 時程與分期 (Timeline / Phasing)

- **MVP 收尾(1–2 週):** 補 **P0-8 即時匯率** + **P0-9 分析**(其餘 P0 已上線)。→ 開始 4 週驗證窗。
- **V1(驗證通過後):** 帳號 + 雲端同步、確認所選(真實紀錄)、分享/匯出、(擇一)聯盟分潤或即時匯率到價提醒。
- **V2:** 轉寄信自動記帳 + 個人偏好檔 + 金錢個人化推薦。

## 附錄:重點取捨 (PoV)

大部分 MVP 已完成。真正還缺、也最關鍵的只有兩件:

1. **把模擬匯率換成即時匯率**(否則護城河「省錢」站不住)。
2. **加最基本的使用分析**(否則無法驗證假設)。

其餘已上線功能中,**回顧 / 心情故事 / 機票住宿估算其實是 P1**,不是驗證核心所必需。MVP 的唯一任務是驗證:**「台灣出境旅客願意用一個先算錢、再排行程的工具」**。
